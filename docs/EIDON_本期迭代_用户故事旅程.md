# EIDON 本期迭代 用户故事旅程

**范围（D-7）：** 本期可交付的完整产品体验——把现有 SoloMD「扁平 Markdown 库」**颠覆性重构（1.x→2.x）**为**固定三层【节点】拓扑 + 多模板 schema**的结构化知识 IDE 内核。既走**新建能力**（节点 + 模板 + 一致性检测），也走**复用现有能力**（编辑 / 保存 / 版本 / 搜索），并显式标注**后续迭代 / 不在范围**。
**重构定位（D-1）：** 不背向后兼容包袱，但**以现有系统为基底、复用优先、最小新增**；颠覆数据模型与产品形态，复用三层【代码】架构。本期同步 **solomd→EIDON 改名**、**磁盘契约统一**；AI·Agent·Recipes **不在 EIDON 范围**（旧代码作基底保留，后续独立重建）。
**不含（后续迭代 / 不在范围）：** 私有快照仓库等「另造快照系统」（**用现有 git 即可**）· 越界 / 孤儿 / 失联软态 · 杂物自动整理 · 自动补全 · 独立一致性面板 · 回收站三关 · **AI·Agent·Recipes 整块（不在 EIDON 范围）** · Todo / 提醒 / 双向链接（除非已存在可超前）/ 语义搜索。
**依据：** `temp/` 三份 EIDON 愿景文档（产品层数据层 ADR / 基座 PRD / 用户故事旅程）+ `eidon-base-roadmap/`（D-1~D-12 决策）+ 工程层 ADR-0011~0018 + 本期 PRD（`EIDON_本期迭代_PRD.md`）。
**主角：** 林（化名）——一名同时做科研、工程、个人知识分享的多领域单人工作者，重视数据本地可控、可迁移、可用 Git 管理。
**组织方式：** 按林从「打开现有库」到「把它改造成结构化」的真实时间线编排，每个故事标注对应 PRD 功能编号，并标注 **【新建】/【复用】/【后续】**。
**日期：** 2026-06-06

## 「三层」消歧（贯穿全文，禁裸用）

- **三层【代码】** = `src/core/src-tauri` 软件分层，本期**不动**。
- **三层【节点】** = `L1/L2/L3` 数据层级，本期**要造**。

本旅程出现的「三层」一律指**三层【节点】**，除非显式写「三层【代码】」。

## 旅程地图（七幕）

```
第一幕 初见       → 打开现有库、首次用结构功能、理解叠加了什么
第二幕 立规矩     → 先建模板（必须先于建结构）—— 保留断点①
第三幕 搭骨架     → 硬强制按深度=层级建 L1/L2/L3（零妥协）
第四幕 填血肉     → 在 L3 写内容、填字段、（复用）自动保存
第五幕 时光机     → （复用）自动快照、diff、回退
第六幕 改造存量   → 把扁平库改造成结构化：违规被标记 → 提升为节点/手动整改 —— 断点②变形
第七幕 信任它     → 离线、迁移、删缓存重建、自有 Git
```

> 与 temp/ 原八幕的差异：原「第六幕 重整理（回收站/越界/孤儿）」+「第七幕 守边界（杂物自动整理/自动补全/一致性面板）」在本期收紧为**第六幕「改造存量」**——只保留「硬强制 + 违规标记 + 手动整改（提升为节点）」，砍掉自动化与软态身份系统（D-6/D-8/D-9）。

## 第一幕：初见（打开现有库与首次启用结构）

### US-001　打开我现有的本地文件夹，编辑体验照旧【复用】
- **作为** 林，**我想** 像以前一样把本地任意文件夹作为工作区打开、照常编辑，**以便** 重构升级不打断我熟悉的编辑/搜索/版本体验。
- **验收：** 复用现有打开流程与编辑栈；纯扁平 Markdown 使用方式**不触碰** `.eidon/`，**不破坏我的 Markdown 内容文件**。（PRD FR-WS-1 / FR-EDIT-1【复用】）
- **说明（2.0 重构）：** 这是一次颠覆性重构（产品已更名 **EIDON**），**不承诺旧 `.solomd` 数据/旧形态的向后兼容**；但「不删改我的内容文件」始终成立——颠覆的是结构与产品形态，不是我的文字。

### US-001a　产品更名 EIDON，AI/Recipes 入口暂时不见了【新建·改名 + 移出范围】
- **作为** 林，**我想** 知道升级后这个工具叫什么、为什么原来的 AI 自动化面板不见了，**以便** 我对这次大改有清晰预期。
- **验收：** 品牌、窗口标题、关于页统一为 **EIDON**（packageName/i18n 全量改名）；原 **AI·Agent·Recipes** 整块**不在 EIDON 范围**（入口不挂载、旧代码作基底保留），后续作为独立的「AI」重建回归。（PRD §12 / 工程层 ADR-0017/0018【新建·改名 + 移出范围】）
- **说明：** 「不在范围 ≠ 删除」——我的历史 Recipe 文件（旧 `.solomd/`）静置不动。

### US-002　首次启用结构功能时，懒初始化并送我一套默认模板【新建】
- **作为** 林，**我想** 第一次用「建模板 / 建节点」时系统才创建 `.eidon/` 并给我「档案 / 项目 / 资料」默认模板，**以便** 我不被迫一打开就接受一套结构，按需启用。
- **验收：** 首次使用结构功能时创建 `.eidon/templates/` + 写内置默认模板（普通文件）；这些模板与我之后自建的完全平级、可改可删。（FR-WS-1 / FR-TPL-3【新建】）

### US-003　再次打开不重复初始化、不塞回我删过的东西【新建】
- **作为** 林，**我想** 第二次打开同一 workspace 时不要重新初始化，**以便** 我删掉的默认模板不会又冒出来。
- **验收：** `.eidon/` 已存在则不重复初始化、不重写内置模板；删掉的内置模板重开后不再自动出现。（FR-WS-1【新建】）

## 第二幕：立规矩（模板必须先于结构）—— 保留断点①

### US-004　在建任何结构之前，先到「设置」定义我的三层【节点】模板【新建】
- **作为** 林，**我想** 在创建第一个顶层结构之前，先到**设置里的模板 / Schema 管理**定义「我这套结构三层各叫什么、各有哪些字段」，**以便** 我的结构用我自己领域的语言（研究方向 / 课题 / 资料），而不是被迫套用别人的词。
- **验收：** **设置面板内**可创建新模板（新建 schema 规范）；创建时依次定义 L1/L2/L3 三层的名字与字段；**未建/未选任何模板时无法创建 L1**——系统引导我「先到设置创建或选择一个模板」。（FR-TPL-1 / FR-NODE-1【新建】）
- **说明（断点①）：** 这是有意的顺序约束——结构由 schema 定义（Schema Driven），所以「先有模板，后有结构」。

### US-005　给每层配置字段，但只用最基础的几种类型【新建】
- **作为** 林，**我想** 给「课题」层加上「研究方向、经费、进度」这些字段，**以便** 我的每个课题都有统一的结构化信息。
- **验收：** 字段类型可选 6 种：文本、多行文本、数字、日期、单选、布尔；每层可加多个字段、设默认值 / 必填 / 选项。（FR-TPL-1【新建】）
- **边界：** 本期没有「关联 / 货币 / 多选 / 评分」等高级类型——后续再加。

### US-006　并列拥有多套模板，对应我的多个领域【新建】
- **作为** 林，**我想** 同时拥有「科研模板（研究方向 / 课题 / 资料）」和「开发模板（产品线 / 特性 / 任务）」两套并列模板，**以便** 在同一知识库里既做科研又做开发，各用各的结构。
- **验收：** 可创建多套模板平级共存；建 L1 时选择用哪套模板。（FR-DATA-2 / FR-TPL-1【新建】）

### US-007　完善模板时，老内容不会被打乱【新建】
- **作为** 林，**我想** 用了一阵后给「课题」模板再加一个字段，而我已建的那些课题保持原样，**以便** 随时完善模板而不必担心破坏既有数据。
- **验收：** 编辑已被使用的模板生成新版本；已有节点保持旧版、字段不变；UI 明确提示「改动只对新建节点生效，已有 N 个可一键批量升级」。（FR-TPL-2【新建】）

## 第三幕：搭骨架（硬强制按三层建结构，零妥协）

### US-008　在工作区根创建第一层【新建】
- **作为** 林，**我想** 用我的科研模板在根创建一个 L1「深度学习研究」，**以便** 开启一个研究方向。
- **验收：** 在根「新建」只给「新建 L1 节点（选模板）」；生成 `.node/node.json`（含 id/templateId/level=1/type/字段）+ 空 README.md + AGENTS.md（占位）。（FR-NODE-1 / FR-DATA-4【新建】）

### US-009　层级由位置决定，我一看就懂在第几层【新建】
- **作为** 林，**我想** 知道「放在根第一层就是 L1、第二层就是 L2、第三层就是 L3」，**以便** 我对结构有绝对清晰、不会搞错的心智模型。
- **验收：** 深度=层级铁律：第 1/2/3 层节点目录分别是 L1/L2/L3，第 4 层起是自由文件夹 / 文件。（FR-DATA-2【新建】）

### US-010　逐层建到第三层，越级路径根本不存在【新建·硬强制】
- **作为** 林，**我想** 在「深度学习研究」下建 L2「小样本学习」、再在其下建 L3「实验记录」，而**系统从 UI 上就不给我任何越级创建的口子**，**以便** 我永远不可能搭出不合规的结构。
- **验收：** L2 只能建在 L1 下、L3 只能建在 L2 下；UI 按当前位置只暴露合法的「新建节点」入口，**不存在**越级创建路径（零妥协）；L2/L3 继承父链所属模板，按对应层 schema 填字段。（FR-NODE-1 / FR-MARK-1【新建·硬强制】）

### US-011　第三层以下我可以随意组织，系统不管【新建】
- **作为** 林，**我想** 在 L3「实验记录」里随便建子文件夹（如「原始数据/」）放文件，**以便** 在资料内部自由组织而不受结构约束。
- **验收：** L3 以下的目录是自由文件夹（无身份、无 `.node/`），系统不赋予节点语义、不做约束。（FR-DATA-2 / FR-TREE【新建】）

### US-012　每层可以写一段它自己的说明【新建】
- **作为** 林，**我想** 给每个节点写一段它自身的描述，**以便** 记录「这个研究方向 / 课题 / 资料是关于什么的」。
- **验收：** 每个节点目录有一个 `README.md` 供我填写人类描述；`AGENTS.md` 为占位（后续 AI 用，本期不展示编辑入口或仅占位）。（FR-DATA-4【新建】）

## 第四幕：填血肉（内容、字段、保存）

### US-013　在资料层写 Markdown 笔记，且只有 L3 能放内容【新建·硬约束 + 复用编辑器】
- **作为** 林，**我想** 在 L3「实验记录」里直接创建并编辑 Markdown 文件（如「实验1.md」），而 L1/L2 根本不给我放内容文件的入口，**以便** 内容只可能出现在 L3，心智零歧义。
- **验收：** 「新建 Markdown / 内容文件」入口**只在 L3 及其自由子文件夹**出现；L1/L2 不提供。编辑**复用现有** CodeMirror（编写 / 预览 / 语法高亮）。（FR-NODE-1 / FR-DATA-4【新建·硬约束】 + FR-EDIT-1【复用】）

### US-014　编辑器随文件大小智能调整【复用·按现状】
- **作为** 林，**我想** 编辑普通笔记时流畅，偶尔打开很大的文件也不崩溃。
- **验收：** **复用现有**编辑器能力；若现有已具备大文件分级（≤2MB 全功能 / 2–10MB 基础 / >10MB 只读）则沿用；**若现有未具备，本期不硬性承诺，标注为后续工程调优。**（FR-EDIT-1【复用·标注】）

### US-015　填写节点的结构化字段【新建】
- **作为** 林，**我想** 给「小样本学习」这个课题填上「经费=50万、进度=进行中」，**以便** 课题有可被检索的结构化属性。
- **验收：** 节点字段按其模板版本对应的字段表单渲染（6 种类型控件）；保存写入 `node.json` 的 fields。（FR-NODE-4 / FR-DATA-5【新建】）

### US-016　我不用记得保存，它自动保命【复用】
- **作为** 林，**我想** 写东西时不必时刻按保存，**以便** 我专注内容，且断电 / 崩溃也不丢太多。
- **验收：** **复用现有**失焦自动保存（`autoSaveDirtyTabs` on blur）；异常退出最多丢失最近一次保存间隔内的编辑。（FR-VER-1 / NFR-2【复用】）

### US-016b　在 L3 里随手放 PDF 或图片，能基础预览或交给系统【复用】
- **作为** 林，**我想** 在 L3 里把 PDF 报告和实验截图直接放进去，并能基础预览或用系统默认程序打开。
- **验收：** **复用现有**非 Markdown 文件基础预览 / 系统打开能力；本期不提供这些类型的编辑。（FR-EDIT-3【复用】）

## 第五幕：时光机（版本与回退）—— 复用现有

### US-017　重要节点产生有意义的版本，而非一堆垃圾记录【复用·autoGit】
- **作为** 林，**我想** 系统在我写到稳定点时自动记录版本，而不是每秒记一条，**以便** 版本历史可读、能找到。
- **验收：** **直接复用现有 git** 的 auto-commit 历史（debounce 自动提交，避免高频垃圾 commit）+ `history.commitNow` 手动立即提交；`core/snapshots` 仅薄封装。**autoGit 无需修改，本期不实现任何快照功能。**（FR-VER-2【复用·git】）
- **消歧：** 这里的 git 版本 = 普通编辑的版本历史，**不是** Agent Recipes 的 AutoGit 分支沙箱（不在 EIDON 范围）。

### US-018　查看历史、对比、回退【复用】
- **作为** 林，**我想** 查看一个文件的历史版本、对比差异、回退到某一版，**以便** 找回误改或对照修改。
- **验收：** **复用现有** HistoryPanel：浏览版本、diff 对比、恢复（恢复语义以现有实现为准）。（FR-VER-3【复用】）

### US-019　版本历史不抢我的工作目录（现状说明 + 后续目标）【复用·标注后续】
- **作为** 林，**我想** 版本历史尽量不污染我的工作目录、不抢我自己的 Git。
- **验收（本期）：** **直接用现有 git**（`core/snapshots` 仅薄封装，autoGit 不改、不做快照功能）；本期**不新建** `.eidon/snapshots.git` 私有仓库。
- **标注后续迭代：** 如未来确需，在 `core/snapshots` 内换实现为独立私有 `snapshots.git`（git-dir/work-tree 分离、workspace 根不出现 `.git/`）+ 保存/快照解耦 + 路径↔ID 补偿（产品层 R-2），不动调用方。（FR-VER-2【复用·标注后续】）

## 第六幕：改造存量（违规标记 + 提升为节点）—— 断点②变形

> 本幕是本期**与现有扁平 SoloMD 库衔接的核心价值**：把一个已经堆满普通文件夹的库，由用户主动「改造」成结构化，而系统**绝不在背后擅自动手**。

### US-020　移动和重命名不会断掉节点身份【新建】
- **作为** 林，**我想** 把一个课题改名、或移到另一个研究方向下，**以便** 灵活调整结构而不丢失它的身份。
- **验收：** 重命名改目录名、移动改父级，节点 ID 不变；id→path 索引更新。（FR-NODE-2 / FR-DATA-3【新建】）

### US-021　跨领域放置被允许，越界只是被标记而不强改【新建】
- **作为** 林，**我想** 把一个「开发特性」挂到某个「研究方向」下（因为这个研究催生了工程实现），**以便** 表达真实的跨领域关联。
- **验收：** 允许跨模板移动；节点字段原样不动；若造成「层级不符 / 位置不当」，**只在 FileTree 打违规标记**（不写软态 `flags`、不进一致性面板——本期收紧），我可选择接受现状或手动移回。（FR-NODE-2 / FR-MARK-2【新建】）

### US-022　我能在文件树里一眼看出哪里不合规【新建·断点②】
- **作为** 林，**我想** 在 FileTree 里直接看出「哪个普通文件夹该提升成节点 / 哪个 L1/L2 里混进了内容文件 / 哪个节点层级不符 / 哪个节点系统文件缺失」，**以便** 我不必非得打开某个面板才知道哪里需要处理。
- **验收（FR-MARK-2 四类标记）：**
  1. **前三层的普通文件夹** → 标记「待提升 / 非节点」；
  2. **L1/L2 里的内容文件** → 标记「位置非法 / 待下沉」；
  3. **节点物理深度与 level 不符** → 标记「层级不符」；
  4. **节点 `.node/` 缺失或损坏** → 标记「结构待修复」。
- **共通约束：** 标记**只读、不阻塞使用、不自动处理**；点击徽标弹出整改入口。本期**不做**独立一致性面板（与原 temp/ §FR-SYNC-3 的差异，见 PRD §0.5）。（FR-MARK-2 / FR-TREE【新建·断点②变形】）

### US-023　把普通文件夹「提升为节点」，按深度自动定层级【新建·整改主入口】
- **作为** 林，**我想** 把现有扁平库里的一个普通文件夹一键「提升为节点」，**以便** 把存量目录逐步改造成结构化，而不必从零重建。
- **验收：** 「提升为节点」按其**当前物理深度**提升——根下 → L1（选模板）、L1 下 → L2、L2 下 → L3（继承父链模板）；生成新 ULID + `.node/node.json`（父链模板对应层字段框架，值由我补）+ 空 README/AGENTS；仅对合法深度（≤3）提供；**系统不自动提升，我点才动。**（FR-NODE-3【新建】）

### US-024　L1/L2 里混进的内容文件，引导我下沉到 L3【新建】
- **作为** 林，**我想** 当我（在外部）不小心把一个文件放进 L1/L2 这种纯组织层时，系统提醒我而不是擅自挪走，**以便** 我的文件位置始终由我掌控（位置本身就是我的数据）。
- **验收：** 默认标记「位置非法 / 待下沉」、不自动移动；我点击后才「移动 / 下沉到某个 L3 节点」或自行删除。（FR-MARK-3【新建】）
- **说明（与原 temp/ 的差异）：** 本期**不做**「自动整理开关」「其他 L3 兜底节点」「自动补全」——只做「标记 + 用户手动整改」。这些自动化属后续迭代。

### US-025　删除是直接删除（现状），回收站三关是后续目标【复用·缩水标注】
- **作为** 林，**我想** 删除一个节点。
- **验收（本期）：** **复用现有** FileTree 删除；删除节点 = 删除整个节点目录（含 `.node/`、子树、内容文件）。**当前现状：直接删除，无回收站。**
- **标注后续迭代：** `.eidon/trash` 物理移入 + 三关恢复（路径冲突改名 / orphan 兜底 / schemaVersion 懒迁移 / reference 自动重连）。（FR-DEL【复用·标注后续】）

## 第七幕：信任它（本地、迁移、可重建、Git）

### US-026　完全离线可用【复用】
- **作为** 林，**我想** 在没有网络时照常使用全部结构功能，**以便** 工作不依赖任何在线服务。
- **验收：** 全部本期基座功能离线可用；联网能力（AI/同步/REST 等）随 AI·Agent·Recipes **不在 EIDON 范围**。（NFR-3【复用】）

### US-027　把知识库拷到另一台电脑，结构与内容完整【新建】
- **作为** 林，**我想** 把整个 workspace 文件夹拷到另一台电脑继续用，结构、节点身份、字段、模板都在。
- **验收：** 拷贝 workspace 目录即完整自包含（模板、`node.json`、内容全在文件里）；git 历史属本机缓存不随迁移。（NFR-4 / FR-DATA-1【新建】）

### US-028　删掉缓存数据库，一切照旧可重建【新建·铁律】
- **作为** 林，**我想** 即使删除了 `.eidon/` 索引缓存（或它损坏了），重启后我的结构、身份、字段、搜索都能恢复，**以便** 我确信「文件才是真理，数据库只是加速器」。
- **验收：** 删除索引后重启，遍历目录（识别 `.node/` → 读 `node.json` → 读 `.eidon/templates/`）100% 重建结构树 / 身份 / 字段 / id→path；重建可后台进行不阻塞使用。（FR-DATA-1 / NFR-6【新建·铁律】）

### US-029　搜索找得到东西【复用 + 可选结构过滤】
- **作为** 林，**我想** 按内容全文搜索、最好也能按标题 / 模板 / 层级 / 字段值做结构化搜索，**以便** 在大知识库里快速定位。
- **验收：** **复用现有** `GlobalSearch` 全文搜索；**可选增强（本期可做可不做）**在其上叠加「按节点 / 模板 / 层级 / 字段」结构过滤。（FR-SEARCH【复用 + 可选】）
- **边界：** 语义 / 向量搜索属第二步 AI，不在本期。

### US-030　我可以用自己的 Git 管理这个知识库【复用·现状说明】
- **作为** 林，**我想** 在 workspace 用我熟悉的 Git 做版本管理，**以便** 用我熟悉的工具掌控历史。
- **验收（本期）：** 复用现有 git 能力；`.eidon/` 索引缓存可被加进 `.gitignore`。
- **标注后续迭代：** 「EIDON 私有快照仓库」与「用户自有 Git」彻底分离（独立 `snapshots.git`，workspace 根不被 EIDON 占用）。（FR-VER-2【复用·标注后续】）

## 本期完成的标志（与 PRD DoD 对应）

林能够：（启用结构）打开现有库 → **在设置内**建模板 → **硬强制**按三层搭 L1/L2/L3 → 在 L3 写内容填字段 →（复用）自动保存与 **autoGit** 版本回退 → 把存量普通文件夹**提升为节点**、违规被 FileTree 标记（`core/consistency`）并手动整改 →（复用）搜索 → 拷走能用、断网能用、删缓存能重建；全程品牌已是 **EIDON**。
