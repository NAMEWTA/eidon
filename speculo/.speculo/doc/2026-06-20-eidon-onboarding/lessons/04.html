1. 节点拓扑：深度=层级铁律

这是 EIDON 最核心的数据模型。理解它，整个产品逻辑就通了：

workspace/                          ← 受管根
├── .eidon/                         ← 系统区（模板+索引）
│   └── templates/                  ← 模板 schema
│
├── 项目/                           ← L1 节点（第1层，纯组织层）
│   ├── .node/node.json             ← 身份文件
│   │   { id:"ulid...", level:1, templateId:"project", fields:{...} }
│   │
│   └── 前端/                       ← L2 节点（第2层，纯组织层）
│       ├── .node/node.json
│       │   { id:"ulid...", level:2, ... }
│       │
│       └── 需求文档/               ← L3 节点（第3层，唯一内容层）
│           ├── .node/node.json     ← 身份文件
│           ├── spec.md             ← 内容文件（.md/.png/...）
│           ├── 草图.png
│           └── 截图/               ← 第4层起：自由文件夹（无身份）
│               └── v1.png
│
└── 个人/                           ← 另一个 L1 节点
    └── .node/node.json
规则详情
深度=层级铁律第1层节点=L1，第2层=L2，第3层=L3。物理位置唯一确定层级
L1/L2纯组织层只含 .node/ + README.md + AGENTS.md + 下级节点目录。禁止放内容文件（.md/.png等）
L3唯一内容层可放任何类型的内容文件 + 自由子目录
第4层起自由普通文件夹，无 .node/，无身份。系统不管
创建期硬强制只能根→L1(选模板)、L1下→L2、L2下→L3。UI不提供越级创建入口
ULID身份每节点唯一ULID，存在.node/node.json，随目录移动（rename/mv不丢身份）

2. node.json 结构（磁盘契约）

// 示例：L3 节点 "需求文档" 的 .node/node.json
{
  "id": "01JQXYZ...",          // ULID，全局唯一
  "templateId": "project",     // 关联的模板
  "level": 3,                  // 层级（物理深度唯一确定）
  "type": "document",          // 节点类型
  "schemaVersion": 1,          // 模板版本号（懒迁移）
  "createdAt": "2026-06-20T...",
  "fields": {                  // 扩展字段（模板定义）
    "status": "进行中",        // select 类型
    "priority": 3,             // number 类型
    "deadline": "2026-07-01"   // date 类型
  },
  "references": [],            // 预留：跨节点引用（本期空）
  "flags": {}                  // 预留：软态标记（本期空）
}

契约定义在 shared/contracts/node.ts（zod schema），对应类型在 shared/models/node.ts。

3. 扫描建树全流程（代码走读）

以 nodes:scan 为例，看看实际代码怎么走：

// ① shared/ipc/channels.ts — 通道声明
'nodes:scan': { req: { root: string }; res: NodeTree }

// ② bridge/ipc/nodes.ts — 前端调用
scan: (root: string) => eidonInvoke('nodes:scan', { root })

// ③ frontend/stores/nodes.ts — Store 调用
const tree = await nodesBridge.scan(root)
// → 写入 Zustand store，触发 FileTree 渲染

// ④ backend/ipc/handlers/nodes.handlers.ts — Handler 接线
ipcMain.handle('nodes:scan', async (_e, req) => {
  return nodeService.scan(req.root)
})

// ⑤ backend/services/node-service.ts — 编排
async scan(root: string): Promise<NodeTree> {
  const store = this.createStore(root)  // 构造注入端口
  return this.domain.scan(store, root)
}

// ⑥ backend/domain/nodes.ts — 业务规则（核心逻辑）
function scan(store: NodeStore, root: string): NodeTree {
  const entries = store.readdir(root)      // → capabilities/editor/file-ops
  const nodes: ScannedNode[] = []

  for (const entry of entries) {
    if (!entry.isDir) continue
    const nodeJsonPath = join(entry.path, '.node/node.json')
    if (!store.exists(nodeJsonPath)) continue  // 普通文件夹跳过

    const raw = JSON.parse(store.readFile(nodeJsonPath))
    const parsed = NodeSchema.parse(raw)  // zod 校验
    const depth = pathDepth(entry.path, root)

    nodes.push({
      ...parsed,
      path: entry.path,
      children: depth < 3 ? scan(store, entry.path) : []  // 递归
    })
  }
  return { root, nodes }
}
💡 性能关键：scan() 是递归的、可能扫万级文件。domain 逻辑纯函数化确保可单测；文件 IO 走注入端口，测试时可用 mock store。

4. 创建节点的三道防线

防线 1 — 前端硬拦截：UI 只在合法位置显示「新建节点」按钮。L1→在根下，L2→在 L1 下，L3→在 L2 下。不存在越级创建的入口。
// frontend/components/NodeCreateDialog.tsx 伪代码
function canCreateHere(parentPath: string): boolean {
  const depth = pathDepth(parentPath, root)
  if (depth === 0) return true   // 根下可建 L1
  if (depth === 1) return true   // L1 下可建 L2
  if (depth === 2) return true   // L2 下可建 L3
  return false                    // L3 下只能建自由文件夹
}
防线 2 — domain 层校验：即使前端放过，backend/domain/nodes.ts 的 createNode() 也会再次检查层级合法性、模板是否存在。
防线 3 — consistency 扫描：backend/domain/consistency.ts 定时/手动扫描，在 FileTree 上标记四类违规：①缺.node/待提升 ②L1/L2有内容文件 ③深度≠level ④.node/损坏。

5. 模板系统（节点配套）

模板定义了每层节点可以有哪些字段。关键规则：

  • 模板 = L1+L2+L3 三层字段集捆绑在一起
  • 版本化不可变：编辑模板生成新版本，旧节点保持旧 schemaVersion
  • 6 种字段类型：text / textarea / number / date / select / boolean
  • 字段二分：元字段（id/templateId/level/...系统自动生成，用户不可增减） vs 扩展字段（模板定义，可扩展）
  • 模板管理 UI 在 Settings 面板中
  • 内置种子（档案/项目/资料）仅首次初始化写入，之后与用户模板平级

模板相关通道：templates:list/create/edit/delete/get/listVersions（7 条）。
