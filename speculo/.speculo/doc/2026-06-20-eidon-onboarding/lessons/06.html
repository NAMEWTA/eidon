🛠️ Lesson 06 · 开发规范与扩展指南

耗时：~15 分钟  |  先修：前 5 课全部

🏆 本课胜利：你掌握全部的提交前检查清单、ESLint 机器强制边界、契约改动流程，并能在 30 分钟内完成一个带 IPC 通道的新功能。

1. 提交前铁律

每次提交前必须全部通过：
  1. pnpm lint — ESLint 四层边界 + no-restricted-imports（机器强制）
  2. pnpm typecheck — renderer + node 双 tsconfig 类型检查
  3. pnpm test:core — shared + backend domain 单元测试
  4. 如果改过磁盘契约：加跑 pnpm contracts:check
  5. 如果改过前端组件：加跑 pnpm test:ui

建议在提交前一次性跑完：

pnpm lint && pnpm typecheck && pnpm test:core && pnpm contracts:check

2. ESLint 机器强制边界（必读）

规则在 app/eslint.config.mjs，用 eslint-plugin-boundaries + no-restricted-imports 实现。以下是各层的禁止清单：

你在哪个目录写代码禁止你 import 的东西允许 import
frontend/backend/* (全部), preload/, electronbridge/, shared/, window.eidon
bridge/backend/* (全部), preload/, electron, react, zustandshared/, window.eidon
backend/shell/ 和 ipc/frontend/, bridge/, preload/backend/services, shared/, electron
backend/services/frontend/, bridge/, preload/, shell/, ipc/, electronbackend/domain/, backend/capabilities/, shared/
backend/domain/everything except shared + 注入端口shared/ + 注入端口
backend/capabilities/frontend/, bridge/, preload/, shell/, ipc/, service/, electronshared/, node:*, 库, emit
shared/所有其他目录, react, zustand, electronnpm 纯库（zod, ulid...）
全局@tauri-apps/*（永久禁止）—
最重要：如果你写了 import { something } from '../../backend/...' 在 frontend 文件中 —— pnpm lint 直接报错，CI 拦截。这个规则不是建议，是硬约束。

3. 磁盘契约改动流程

改 .node/node.json 或 template 或 .eidon/ 目录布局形状时，必须严格按此流程：

Step 1: 改 shared/contracts/node.ts （或 template.ts / todos.ts）
         → 更新 zod schema

Step 2: 更新 fixtures/contracts/ 下对应的 golden fixture
         → fixtures/contracts/node.l3.json

Step 3: 跑 pnpm contracts:check
         → 必须绿！红了 = 破坏扫描重建一致性

Step 4: 改解析器（domain/*.ts 中读 JSON 的逻辑）
         → 适配新 schema 形状

Step 5: 跑 pnpm test:core
         → 确认所有 conformance 测试仍然绿
💡 为什么 fixtures 先于解析器？因为「删缓存→从 .node/ + .eidon/templates/ 100% 重建节点树」是产品层的可重建铁律（AX-1/AX-4）。fixtures 是一道防线——保证任何改了契约形状但没有对应更新的解析逻辑都能被抓出来。

4. 新增功能的完整模板（照着做）

假设要新增一个 「标签系统」（tags:list / tags:getNodeTags / tags:searchByTag）。按以下步骤走：

4.1 声明类型和契约

// ① shared/models/tag.ts（新建）
export interface Tag { name: string; count: number }
export interface TagSearchResult { nodeId: string; path: string; tags: string[] }

// ② shared/ipc/channels.ts（追加）
export interface IpcContract {
  // ... 现有 85 通道 ...

// 补全 CHANNEL_PRESENCE
export const CHANNEL_PRESENCE: Record<keyof IpcContract, true> = {
  // ... 现有 ...

4.3 编排

// ⑤ backend/services/tag-service.ts（新建）
export function createTagService(store: NodeStore) {
  return {
    list: (root: string) => scanTags(store, root),
    getNodeTags: (nodePath: string) => getNodeTags(store, nodePath),
    searchByTag: (root: string, tag: string) => searchByTag(store, root, tag),
  }
}

4.4 接线

// ⑥ backend/ipc/handlers/tags.handlers.ts（新建）
import { registerIpcHandlers } from '../register'
import { createTagService } from '../../services/tag-service'

export function registerTagHandlers(getStore: () => NodeStore) {
  const service = createTagService(getStore())

  ipcMain.handle('tags:list', async (_e, req) => service.list(req.root))
  ipcMain.handle('tags:getNodeTags', async (_e, req) => service.getNodeTags(req.nodePath))
  ipcMain.handle('tags:searchByTag', async (_e, req) => service.searchByTag(req.root, req.tag))
}

// ⑦ backend/shell/index.ts（注册）

6. 测试规范

  • domain 层：纯函数单测，mock 注入端口。写在 backend/domain/__tests__/
  • capabilities 层：纯函数单测，mock node:*。写在各自 __tests__/ 子目录
  • shared 层：单元测试 + 契约 conformance。写在 shared/__tests__/
  • 前端 store/composable：mock bridge 返回值。写在 frontend/__tests__/（如有）
  • E2E：在 backend/domain/__tests__/eidon/e2e.test.ts
💡 最终检验（学完全部 6 课）：选一个真实的小功能（如「新增 shell:getAppVersion 通道」或「新增一个文件导出格式」），从 channels.ts → backend → bridge → frontend 完整走通。遇到任何卡住的地方，回来翻对应课程，或者直接问我。
