Part A：节点级待办与提醒

A1. 数据模型：每个节点独立持有

待办存在每个节点自己的 .node/todos.json 中——随节点迁移、删缓存可重建：

// 某 L3 节点的 .node/todos.json
{
  "items": [
    {
      "id": "todo_ulid_1",
      "text": "完成需求文档初稿",
      "done": false,
      "createdAt": "2026-06-20T10:00:00Z",
      "completedAt": null,
      "reminder": {
        "enabled": true,
        "at": "2026-06-21T09:00:00Z",     // 提醒时间
        "repeat": "daily"                   // none | daily | weekly | monthly
      }
    }
  ]
}

契约定义：shared/contracts/todos.ts（NodeTodoFileSchema），golden fixture：fixtures/contracts/todos.l3.json。

A2. 扫描流程

// 前端触发
todosBridge.scan({ nodeRefs: [{id:"...", path:"..."}] })
// → 'todos:scan' IPC → todo-service → domain/todos.ts

// backend/domain/todos.ts 核心逻辑（简化）
async function scanTodos(store: NodeStore, refs: NodeRef[]): Promise<AggregatedTodo[]> {
  // 批量并行读取（性能铁律）
  const results = await Promise.all(
    refs.map(async (ref) => {
      const todoPath = join(ref.path, '.node/todos.json')
      if (!store.exists(todoPath)) return []
      try {
        const raw = JSON.parse(store.readFile(todoPath))
        const parsed = NodeTodoFileSchema.parse(raw)  // zod 校验
        return parsed.items.map(item => ({ ...item, nodeRef: ref }))
      } catch {
        return []  // 坏 JSON → 回退空数组，不中断
      }
    })
  )
  return results.flat()
}

A3. 提醒调度

前端 frontend/lib/reminder-scheduler.ts 负责调度——纯 setTimeout 模型，不依赖 cron：

// 简化的调度逻辑
class ReminderScheduler {
  private timerId: number | null = null

  schedule(todos: AggregatedTodo[]) {
    const due = todos.filter(t => t.reminder?.enabled && !t.done)
    const earliest = earliestFireAt(due)  // shared/utils/reminders.ts 纯函数

    if (this.timerId) clearTimeout(this.timerId)
    if (earliest) {
      this.timerId = setTimeout(() => {
        // 触发 Electron Notification（通过 bridge/ipc/notification.ts）
        notifyBridge.notify({ title: 'EIDON 提醒', body: earliest.text })
        // 标记已提醒，写回磁盘
        todosBridge.writeNode({ nodeRef: earliest.nodeRef, items: markNotified(...) })
        // 递归调度下一个
        this.schedule(getRemaining(todos))
      }, earliest.at.getTime() - Date.now())
    }
  }
}
💡 设计要点：提醒计算是纯函数（shared/utils/reminders.ts），可在 Node 下单测。调度是前端副作用（composable/lib），不侵入 domain。

Part B：Git 版本管理

B1. 技术方案：isomorphic-git 替代 Rust git2

EIDON 用 isomorphic-git（纯 JS）实现所有 git 操作，替代了旧 Tauri 方案的 Rust git2 crate。17 条 git IPC 通道全部落在 backend/capabilities/git/：

backend/capabilities/git/
├── git-client.ts    # isomorphic-git 客户端封装（init/clone/config）
├── ops.ts            # init/autoCommit/createBranch/checkout/restoreHead
├── history.ts        # fileHistory/fileAtVersion/rollbackFile/dirty
├── diff.ts           # fileDiff
├── prune.ts          # pruneHistory (历史修剪，见 ADR-0023)
└── cloud-folder.ts   # cloudDetect (检测是否在 iCloud/Dropbox 等)

B2. 自动提交流程

// frontend/composables/useAutoCommit.ts
// 每次编辑器失焦保存后触发

async function autoCommit(filePath: string) {
  // ① 调 git:autoCommit 通道
  const result = await gitBridge.autoCommit({
    filePath,
    message: `auto: ${basename(filePath)} — ${new Date().toISOString()}`
  })

  // ② 后端 capabilities/git/ops.ts
  //    isomorphic-git: git.add() → git.commit() → git.log()

  // ③ 如果用户设置了历史上限，触发修剪
  if (settings.historyMaxCommits > 0) {
    const { total } = await gitBridge.repoSize({ root })
    if (total > settings.historyMaxCommits) {
      await gitBridge.pruneHistory({
        root,
        maxCommits: settings.historyMaxCommits
      })
    }
  }
}

B4. 历史修剪（ADR-0023）

为防止 .git/ 无限膨胀，提供修剪能力：

  • 默认关闭：historyMaxCommits = 0（不限），老用户不受影响
  • 用户显式设上限后：每次提交后去抖检查，超限则截断旧提交 + git gc
  • 不可逆但安全：只删 .git/ 缓存（真理源是 plain files），下限 ≥10 防误操作
