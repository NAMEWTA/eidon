2. 线程模型：为什么所有 I/O 都必须 spawn_blocking？

Tauri 的 IPC 线程（invoke handler）是单线程事件循环——所有 #[tauri::command] 默认在此线程执行。如果同步阻塞在文件 I/O 上，整个应用 UI 冻结，Windows 甚至会杀进程。

标准模式

use tauri::async_runtime::spawn_blocking;

/// 异步包装：把 I/O 踢出 IPC 线程
#[tauri::command]
pub async fn read_file(path: String) -> Result<FileReadResult, String> {
    spawn_blocking(move || read_file_inner(path))
        .await
        .map_err(|e| format!("join: {e}"))?
}

/// 同步实现：可独立测试，不依赖 Tauri runtime
pub fn read_file_inner(path: String) -> Result<FileReadResult, String> {
    let bytes = fs::read(&path)...;
    // ... 编码处理 ...
}

关键设计：每个命令拆成两层——async fn command()（Tauri 入口，调用 spawn_blocking） 和 pub fn command_inner()（纯同步逻辑，可脱离 Tauri 单测）。 这个模式在 file_ops、search、git_history、list_dir 中一致使用。

3. 编码检测全链路（file_ops.rs 核心）

EIDON 的 read_file 不是简单的 fs::read_to_string——它实现了完整的 编码检测→解码→回退链路，确保无论文件用什么编码存储，用户都能看到正确内容， 且保存时原编码写回。

read_file 数据流：
fs::read(&path) → sniff_bom(bytes) → chardetng 猜测 → encoding_rs 解码 → had_errors? → lossy UTF-8 fallback → FileReadResult { content, encoding, language, had_bom }

四阶段详解

阶段机制crate作用
1. BOM 嗅探 检查文件头 2-3 字节：EF BB BF→UTF-8、FF FE→UTF-16LE、FE FF→UTF-16BE encoding_rs BOM 是唯一 100% 可靠的编码信号，优先使用
2. 统计检测 EncodingDetector::new().feed(bytes, true).guess() chardetng Mozilla 的字符编码检测库，对无 BOM 文件统计字节模式推断编码（GBK/Big5/Shift_JIS/EUC-KR 等）
3. 严格解码 encoding.decode_without_bom_handling_and_without_replacement() encoding_rs 尝试严格解码——如果文件声明了 GBK 但实际包含非法字节，此步失败
4. 容错回退 encoding.decode() 或 String::from_utf8_lossy() encoding_rs / std 严格解码失败→先试宽松解码；仍有错→UTF-8 lossy（用 � 替换非法字节），绝不报错

write_file：原编码写回

#[tauri::command]
pub async fn write_file(path: String, content: String, encoding: String) -> Result<(), String> {
    // 前端传回 read_file 返回的 encoding 标签（如 "GBK"）
    let enc = Encoding::for_label(encoding.as_bytes()).unwrap_or(UTF_8);
    let (cow, _, had_errors) = enc.encode(&content);
    if had_errors {
        return Err(...);  // 新内容含该编码不可表示的字符
    }
    fs::write(&path, cow.as_ref())...;
    watcher::mark_self_write(&path);  // 抑制 watcher 的 self-write 事件
}
⚠️ 关键问题：如果一个 GBK 编码的 .md 文件包含非法字节（如单字节 >0x7F 不成对）， decode_without_bom_handling_and_without_replacement 失败 → decode() 用 replacement character 替换 → 若仍有问题 → String::from_utf8_lossy() 用 U+FFFD 替换。用户不会看到错误，但可能看到乱码。

4. 文件监听架构（watcher.rs）

文件监听是 EIDON 最精妙的 Rust 模块之一。它解决了三个硬问题：

4.1 原子保存生存（v4.2 关键修复）

VSCode / TextEdit / Vim 的保存模式是 写临时文件 → 重命名覆盖（atomic save）。 如果 notify 直接 watch 文件 inode，重命名会换掉 inode，watch handle 永久失聪。 解决方案：watch 父目录，在回调中按文件名过滤。

4.2 自写抑制

EIDON 自己写文件后，watcher 会在 ~300ms 内收到 OS 文件变更事件。 如果不抑制，会触发「外部修改」对话框——用户每保存一次就被问一次。 通过全局 SELF_WRITES: OnceLock<Mutex<HashMap<String, Instant>>> 记录自写时间戳，回调中检查并抑制 500ms 内的自写事件。

4.3 同步重写窗口

GitHub pull 或 E2EE 解密会批量重写整个 workspace 的文件。 为此提供 mark_workspace_rewrite_window(workspace)，在 30 秒窗口内抑制该 workspace 下所有事件。

架构图

watch_file(path)
  ├─ canonicalize(path)           // 解析符号链接，统一为规范路径
  ├─ inner.watched_files 登记     // 规范路径 → 原始路径（用于 emit）
  ├─ inner.watched_dirs 引用计数   // 目录 → 文件数，为 0 时 unwatch 目录
  ├─ ensure_watcher()             // 懒初始化：首次 watch 时创建 Debouncer
  └─ 父目录 is_new → OS watch    // 同一目录多文件只注册一次 OS watch

notify 事件回调
  ├─ canonicalize(event.path)     // 尝试解析（文件删除则用原始路径）
  ├─ 查 watched_files             // 过滤：只处理 UI 关心的文件
  ├─ 查 SELF_WRITES               // 抑制：500ms 内的自写跳过
  ├─ 查 SYNC_REWRITE_WINDOWS      // 抑制：30s 内的批量重写跳过
  └─ emit("solomd://file-changed", original_path)
