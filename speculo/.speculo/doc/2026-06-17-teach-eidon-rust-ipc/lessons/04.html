🔄 第4课：数据流实战 — 五大真实场景的全链路追踪

目标：把前 3 课的知识串联起来，追踪完整用户场景中前端↔Rust 的数据流向。

场景一：用户打开文件

1. 用户点击左侧文件树中的 hello.md FileTree.tsx → invoke('read_file', { path: '/workspace/hello.md' }) 2. Tauri IPC 序列化参数 → editor/file_ops.rs #[tauri::command] async fn read_file(path: String) → spawn_blocking → read_file_inner() 3. 读原始字节 → 检测编码 (chardetng) → 解码 UTF-8 → 返回 FileReadResult { content: "# Hello…", encoding: "UTF-8", language: "markdown", had_bom: false } 4. 前端收到 → 创建/切换到对应 Tab → CodeMirror 设置内容 useFiles.ts → invoke('watch_file', { path: '/workspace/hello.md' }) 5. watcher.rs → 注册文件到 notify 监控 → 如果目录第一次被监控则注册 OS watch
关键链路：read_file (Command通道) → watch_file (Command通道，注册后续Event通道的基础设施)

场景二：用户编辑并自动保存

1. 用户在 CodeMirror 编辑器中修改文本 Editor.tsx → JS 防抖 (debounce ~1s) 2. 防抖结束 → 触发保存 → invoke('write_file', { path, content, encoding: 'UTF-8' }) 3. Rust: 用指定编码将文本写入磁盘 editor/file_ops.rs → fs::write(path, encoded_bytes) 4. 标记 self-write（关键！） → editor/watcher.rs mark_self_write(path) 在全局 HashMap 中记录：{ "/workspace/hello.md": Instant::now() } 作用：抑制 500ms 内自己的写操作引起的外部变更通知 5. 保存成功后 → AutoGit 自动提交 → invoke('git_auto_commit', { folder, file_path, message }) git/git_history.rs → stage → commit → 返回新 SHA 或 None（无变更时）
自写抑制 (self-write suppression) 机制：这是一个精巧的设计。
① 用户保存文件 → Rust 写磁盘 → watcher 检测到文件变更
② 如果不做抑制，watcher 会立刻通知前端「文件被外部修改了！」
③ 前端弹窗问「外部程序修改了此文件，是否重新加载？」
④ 用户困惑：「我刚自己保存的…」
解决方案：write_file 后，在全局 HashMap 中记录该路径 + 当前时间戳。watcher 回调检查这个 HashMap，如果该路径在 500ms 内被自己写过，就跳过通知。

场景三：外部程序修改了文件

1. VS Code / Obsidian 修改了 hello.md 并保存 2. OS 文件系统发送变更事件 → notify crate 捕获 editor/watcher.rs → debouncer 回调触发（300ms 防抖窗口） 3. 过滤检查： - 该文件的 canonical path 在 watched_files 中吗？ ✅ 是的（场景一注册了） - 该路径在 self-writes map 中且未超 500ms 吗？ ❌ 没有（外部程序写的） 4. 通过检查 → 发送事件 → emit("eidon://file-changed", "/workspace/hello.md") 5. 前端收到事件 useFileWatcher.ts → 对比 payload 路径与当前打开的文件路径 → 匹配！弹出对话框："此文件已被外部修改，是否重新加载？"

场景四：用户打开文件夹 → workspace index 初始化

1. 用户通过 File → Open Folder 选择一个文件夹 Toolbar.tsx / App.tsx → invoke('workspace_index_init', { folder: '/workspace' }) 2. Rust: 全量扫描 knowledge/workspace_index.rs → - 遍历所有 .md/.markdown 文件 (walkdir) - 每个文件：解析 YAML front matter + wikilinks + tags + headings - 存到全局 RwLock<HashMap> 中 - 序列化写入磁盘缓存 (<app_data>/index/<hash>.json) 3. 启动文件监听 → notify watcher 以 Recursive 模式监控整个文件夹 4. 通知前端 → emit("eidon://index-updated", "init") 5. 前端收到 → 更新 UI - 刷新 Wikilink 自动完成列表 - 刷新 Tags 面板 - 刷新 Backlinks 面板
⚠️ workspace_index 使用独立的 notify watcher（不是 watcher.rs 的 watcher）。
watcher.rs 监控的是打开标签页的父目录（NonRecursive），用于检测外部编辑器修改。
workspace_index 监控的是整个 workspace 文件夹（Recursive），用于检测任何 .md 文件的增删改。

场景五：用户试图关闭窗口（有未保存内容）

1. 用户点击 macOS 红色关闭按钮 / Windows X 按钮 2. OS → Tauri 运行时：CloseRequested 事件 runner.rs → RunEvent::WindowEvent { CloseRequested } 3. 检查 FORCE_CLOSE 标志 - 如果 true → 允许关闭（前端已确认） - 如果 false → 阻止关闭 + 通知前端 api.prevent_close() → emit("eidon://close-requested", ()) 4. 前端收到事件 → 检查所有打开标签页的 dirty 状态 App.tsx → - 如果有未保存的标签页 → 弹出确认对话框 - 用户点「放弃并关闭」→ invoke('force_close_window') 5. Rust 收到 force_close_window runner.rs → FORCE_CLOSE.store(true) → window.close() 6. 这次 CloseRequested 再触发，FORCE_CLOSE=true，允许通过
