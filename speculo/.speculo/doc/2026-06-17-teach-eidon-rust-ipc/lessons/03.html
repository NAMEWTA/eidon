📡 第3课：通道二三四 — Events、Plugins、Menu

目标：理解 Rust→前端的 3 种推送通信方式，以及 Menu 事件如何工作。

一、四种通信方式全景对比

方式方向有无返回值EIDON 中用在哪
① Tauri Command
invoke('cmd', args)		 前端 → Rust ✅ 有（Promise） 读写文件、搜索、git操作、索引…（所有功能）
② Tauri Event
emit("event", payload)		 Rust → 前端 ❌ 无（单向推送） 文件外部变更通知、索引更新、菜单点击、窗口关闭询问
③ Plugin API
（封装好的命令+事件）		 双向 依插件而定 原生对话框、剪贴板、通知、窗口状态
④ Menu Event
原生菜单点击		 Rust → 前端 ❌ 无 File/Edit/View/Help 菜单 → App.vue dispatchMenuAction
⑤ Dev Bridge
（仅 debug）		 外部 → Rust → WebView ✅ 有 eidon-dev-mcp 驱动 UI 测试

二、通道②：Event 事件 — Rust 主动推送前端

Command（前端→Rust）
invoke() → 返回 Promise
前端主动调用，等待响应
Event（Rust→前端）
Rust 主动推送，前端被动收听
没有返回值，fire-and-forget

EIDON 中的所有 Event

事件名发送位置（Rust）接收位置（前端）触发场景
eidon://menu runner.rs on_menu_event App.tsx → dispatchMenuAction 用户点击原生菜单项
eidon://close-requested runner.rs CloseRequested App.tsx 用户点窗口关闭按钮
eidon://file-changed watcher.rs 回调 useFileWatcher.ts 外部程序修改了已打开的标签页文件
eidon://index-updated workspace_index.rs handle_event workspaceIndex store 文件创建/修改/删除后重新索引完成

sender 端（Rust）

// ===== 方式 A：在 Setup 中发事件 =====
// runner.rs
.on_menu_event(|app_handle, event| {
    let id = event.id().0.clone();
    let _ = app_handle.emit("eidon://menu", id);
    // emit 返回 Result，用 let _ =  忽略（事件发送失败不致命）
})

// ===== 方式 B：在命令中发事件 =====
// workspace_index.rs
fn workspace_index_init_inner(app: AppHandle, folder: String) -> Result<usize, String> {
    // ... 扫描索引 ...
    let _ = app.emit("eidon://index-updated", &"init");
    Ok(count)
}

// ===== 方式 C：在 watcher 回调中发事件 =====
// watcher.rs — 文件系统变更回调（在 notify 的线程中）
let _ = app_handle.emit("eidon://file-changed", original_path);

receiver 端（前端）

// ===== App.tsx — 监听菜单和关闭事件 =====
import { listen } from '@tauri-apps/api/event';

useEffect(() => {
  const unlistens: UnlistenFn[] = [];

  listen<string>('eidon://menu', (event) => {
    const menuId = event.payload;  // "file.save" / "view.toggleTheme" …
    dispatchMenuAction(menuId);
  }).then(fn => unlistens.push(fn));

  listen('eidon://close-requested', () => {
    checkUnsavedTabs();
  }).then(fn => unlistens.push(fn));

  return () => { unlistens.forEach(f => f()); };  // 清理
}, []);
⚠️ 注意： listen() 返回一个 Promise<UnlistenFn>，需要在组件卸载时调用 unlisten() 取消监听，否则内存泄漏。

三、通道③：Plugin 插件

Plugin 本质上是 "预制的 Tauri Command 集合"。社区为常见需求（文件对话框、剪贴板、通知…）写了现成的 Rust 代码和前端的 JS wrapper，你只需安装和初始化即可使用。

EIDON 使用的插件一览

插件Cargo.toml 依赖初始化位置前端使用方式
Dialog 对话框 tauri-plugin-dialog .plugin(tauri_plugin_dialog::init()) import { open } from '@tauri-apps/plugin-dialog'
Clipboard 剪贴板 tauri-plugin-clipboard-manager .plugin(tauri_plugin_clipboard_manager::init()) import { writeText } from '@tauri-apps/plugin-clipboard-manager'
Notification 通知 tauri-plugin-notification .plugin(tauri_plugin_notification::init()) 定时器提醒：到点弹原生系统通知
Opener 打开文件 tauri-plugin-opener .plugin(tauri_plugin_opener::init()) 用默认程序打开文件/URL
Window State tauri-plugin-window-state .plugin(tauri_plugin_window_state::Builder::default()...) 自动保存/恢复窗口位置、大小、最大化状态
插件的通信原理：插件内部也是通过 #[tauri::command] + emit 来实现通信的。比如 dialog 插件暴露了 open() 命令让前端打开文件选择器，然后用事件把结果传回来。你安装的插件会自动注册到应用中，无需手动写 generate_handler!。

四、通道④：Menu Event — 原生菜单 → 前端

// 1. Rust 构建多语言菜单（runner.rs::build_app_menu()） let save = MenuItemBuilder::with_id("file.save", s.save) .accelerator("CmdOrCtrl+S") .build(app)?; // 2. 注册全局菜单事件转发器 .on_menu_event(|app_handle, event| { let id = event.id().0.clone(); // "file.save" let _ = app_handle.emit("eidon://menu", id); // ↑ 用 Event 通道转发给前端 }) // 3. 前端收到 event.payload → dispatchMenuAction("file.save") // → 触发编辑器保存当前文件

关键设计：Menu Event 本质上复用了 Event 通道。Rust 不直接执行业务逻辑，只是把菜单项 ID 转发给前端，由前端 JS 决定做什么。这保持了「前端拥有所有业务逻辑」的架构原则。

五、通道⑤：Dev Bridge（仅 debug）

这是一个仅 debug 构建可用的 HTTP JSON-RPC 服务器，监听 127.0.0.1 的随机端口。
用途：外部工具（eidon-dev-mcp）可以通过 HTTP POST /eval 来在 WebView 中执行 JS 并获取返回值。 
// 外部工具发送 HTTP 请求
POST /eval
Authorization: Bearer <token>
{ "script": "document.querySelector('.editor').innerText" }

// Bridge 通过 WebviewWindow::eval() 在 WebView 中执行
// WebView 执行后 fetch POST 回 /result/<id>
// Bridge 把结果返回给外部调用者

→ { "ok": true, "value": "# Hello World\n\nThis is a note." }
⚠️ 仅 debug_assertions 编译：Release 构建中这个模块完全不存在，nm 命令检查二进制应得零符号。
