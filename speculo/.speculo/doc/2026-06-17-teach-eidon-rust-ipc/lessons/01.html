二、两个入口：为什么有 main.rs 和 lib.rs？

核心概念：Tauri 2 应用有两种编译模式：
  • 桌面端（macOS/Windows/Linux）：从 main.rs 进入 → 调用 runner::run()
  • 移动端（iOS/Android）：从 lib.rs 进入 → #[cfg_attr(mobile, tauri::mobile_entry_point)] 标记的 run()
两个入口的 命令注册列表几乎完全相同，但 runner.rs 多了桌面菜单、窗口管理等桌面专属逻辑。

三、启动流程（桌面端）

// 1️⃣ main.rs — 创建 Tokio 异步运行时，然后交给 runner let rt = tokio::runtime::Builder::new_multi_thread() .enable_all() .build(); runner::run(); // 2️⃣ runner.rs::run() — 构建 Tauri 应用 tauri::Builder::default() // 安装内置插件（对话框、剪贴板、通知…） .plugin(tauri_plugin_dialog::init()) .plugin(tauri_plugin_clipboard_manager::init()) // 注册共享状态 .manage(watcher::WatcherState::new()) // ⭐ 注册所有前端可调用的命令 .invoke_handler(tauri::generate_handler![ file_ops::read_file, file_ops::write_file, // ... 40+ 个命令 watcher::watch_file, watcher::unwatch_file, ]) // 原生菜单点击→转发给前端 .on_menu_event(|app_handle, event| { app_handle.emit("eidon://menu", event.id().0); }) // 启动时：显示窗口 + 启动调试桥 .setup(|app| { ... }) // 3️⃣ 构建并运行事件循环 .build(tauri::generate_context!()) .run(|app_handle, event| { // 运行循环：监听窗口关闭、Ready 等事件 });
⚠️ 关键发现：lib.rs 和 runner.rs 都各自有 run() 函数！
runner.rs 的 run() 是桌面端 实际在用 的。
lib.rs 的 run() 是给移动端的，但代码里 main.rs 调的是 runner::run()。
二者的命令列表基本一致，但 runner.rs 多出 force_close_window、set_menu_language、convert::convert_file_to_markdown 等命令。

四、模块路径的特殊声明方式

注意 EIDON 使用 #[path = "..."] 来声明模块：

// 不用传统的 mod.rs 嵌套目录，而是这样：
#[path = "editor/file_ops.rs"]
pub mod file_ops;

#[path = "knowledge/search.rs"]
pub mod search;

这样做的好处是保持文件树扁平化，每个 .rs 文件路径直接反映其逻辑归属（editor/、knowledge/、git/），而模块名保持简短。

五、平台条件编译

不是所有代码都在所有平台编译。EIDON 大量使用 Rust 的条件编译属性：

属性含义受影响的模块
#[cfg(not(target_os = "android"))] Android 上不编译 git_history, git_ops, git_prune, cloud_folder
#[cfg(debug_assertions)] 仅 debug 构建编译 dev_bridge
#[cfg(desktop)] 仅桌面端 print_webview, window_state 插件
#[cfg(target_os = "macos")] 仅 macOS apply_macos_language, AppKit 菜单
